# Laravel Directory Structure – Documentation

## 1. app/

The **app** directory contains the core application code. This is where most of the business logic of the Laravel application is written.

It typically includes:

- Controllers
- Models
- Middleware
- Jobs
- Events
- Policies
- Console commands

Example structure:

app/\
├── Http/Controllers\
├── Models\
├── Middleware

Purpose:

- Organizes the main logic of the application.
- Keeps application code separate from framework code.

---

## 2. bootstrap/

The **bootstrap** directory is responsible for bootstrapping the Laravel framework. It initializes the application and prepares it to run.

Important file:

- `bootstrap/app.php`

It also contains:

- `bootstrap/cache` → stores cached framework files like configuration, routes, and services.

Purpose:

- Initializes the Laravel application.
- Loads the framework core components.

---

## 3. config/

The **config** directory stores all configuration files for the application.

Each file manages a specific configuration area.

Examples:

- `app.php` → application configuration
- `database.php` → database connection settings
- `mail.php` → mail server configuration
- `cache.php` → caching configuration

Purpose:

- Central location for application configuration.
- Makes it easy to manage and modify settings.

---

## 4. database/

The **database** directory contains files related to database management.

It includes:

- **Migrations** → define database table structure
- **Seeders** → insert sample or initial data
- **Factories** → generate fake data for testing

Example structure:

database/\
├── migrations\
├── seeders\
├── factories

Purpose:

- Manage database schema and data.

---

## 5. public/

The **public** directory is the web server root. It contains files that are accessible directly from the browser.

Important contents:

- `index.php` → main entry point of the application
- CSS files
- JavaScript files
- Images

Purpose:

- Exposes only necessary files to the web.
- Protects application files from direct access.

---

## 6. resources/

The **resources** directory stores raw frontend resources that are used by the application.

It contains:

- Blade templates (views)
- CSS files
- JavaScript files
- Language files

Example structure:

resources/\
├── views\
├── css\
├── js\
├── lang

Purpose:

- Store frontend files before they are compiled or processed.

---

## 7. routes/

The **routes** directory contains all the route definitions for the application.

Routes define how URLs connect to controllers or logic in the application.

Common route files:

- `web.php` → routes for web interface
- `api.php` → routes for APIs
- `console.php` → routes for console commands
- `channels.php` → routes for broadcasting

Example route:

```php
Route::get('/users', [UserController::class, 'index']);
```

Purpose:

- Handle URL requests and map them to application logic.

---

## 8. storage/

The **storage** directory contains files generated by the application.

It stores:

- Log files
- Cached data
- Compiled Blade templates
- Uploaded files

Example structure:

storage/\
├── app\
├── framework\
├── logs

Purpose:

- Store runtime-generated data safely.

---

## 9. vendor/

The **vendor** directory contains all third-party packages and dependencies installed using Composer.

This includes:

- Laravel framework files
- External PHP libraries

It is automatically generated when running:

composer install

Important:

- Developers should **not modify files inside the vendor directory**.

Purpose:

- Store external libraries required by the project.

---

# Why public/index.php is the Entry Point

`public/index.php` is the **single entry point for all HTTP requests** in a Laravel application.

When a user accesses the application:

1. The request is sent to the web server (Apache or Nginx).
2. The server directs the request to `public/index.php`.
3. `index.php` loads the Laravel framework.
4. Laravel processes the request using routes, controllers, and middleware.
5. The response is returned to the browser.

Reasons for using a single entry point:

- Improves application security
- Implements the **Front Controller Pattern**
- Prevents direct access to internal application files

Request flow:

Browser Request\
↓\
public/index.php\
↓\
Laravel Application\
↓\
Routes → Controller → Response

---

# What is bootstrap/app.php

`bootstrap/app.php` is responsible for creating and configuring the Laravel application instance.

It performs the following tasks:

- Creates the Laravel application object
- Binds core services into the service container
- Registers service providers
- Prepares the application to handle requests

Simplified startup flow:

Request\
↓\
public/index.php\
↓\
bootstrap/app.php\
↓\
Load Service Providers\
↓\
Execute Routes\
↓\
Controller\
↓\
Response

Purpose:

- Initialize the Laravel framework.
- Prepare the application to process requests.
